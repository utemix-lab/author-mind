# Semantic Engine v1 — движок смыслов с IMU

Semantic Engine v1 — это первая операционализированная версия движка,
которая:

- читает структуру author-mind,
- оценивает смысловые объекты по IMU,
- понимает экономику смыслов,
- предлагает автору осмысленные действия.

Если `semantic-engine.md` описывает концепцию,  
то v1 — **рабочую спецификацию**.

---

## 1. Цели Semantic Engine v1

1. Оценивать смысловые объекты (ideas, lines, worlds, stories, exports) по IMU.  
2. Выстраивать приоритеты эволюции смыслов.  
3. Помогать автору распределять внимание.  
4. Поддерживать внутреннюю Sense Economy.  
5. Работать поверх онтологии и не ломать её.

---

## 2. Основные входы и выходы

### Входы:

- структура файлов (фронтматтер + путь + тип)
- связи (related_lines, themes, worlds, projects)
- история изменений (если доступна)
- плотность упоминаний
- участие в exports/
- “эмоциональные” и “мировые” связи (через themes/worlds)

### Выходы:

- оценка IMU для каждой сущности
- предложения по развитию / сворачиванию линий
- предложения по экспорту
- подсказки автору: *“куда вложить внимание”*

---

## 3. Абстрактная модель расчёта IMU v1

Без привязки к конкретному коду, но с чёткой логикой:

### 3.1 Параметры

Для каждой сущности (например, линии):

- `links_count` — количество связей (themes, worlds, stories, projects)
- `conflict_density` — количество конфликтов/дивергенций, связанных с ней
- `age` — “возраст” (по времени или по количеству сессий)
- `export_count` — сколько раз смысл ушёл в другие репо
- `story_presence` — используется ли в stories/
- `world_impact` — влияет ли на worlds/
- `change_frequency` — как часто меняется
- `entropy_level` — насколько размыт / противоречив

### 3.2 Примерная функция

(Псевдологика — для Codex и будущих агентов):

```text
mass      ~ links_count + conflict_density + export_count + world_impact
velocity  ~ change_frequency + story_presence
potential ~ (conflict_density + world_impact + export_count == 0 ? high : medium)
resonance ~ links_count + story_presence + cross-project-references
entropy   ~ текстовая размытость, противоречивость, отсутствие явных связей

IMU ≈ (mass × velocity × potential × resonance) / entropy
Важно: это не точная формула, а архитектурная модель.

4. Поведение Semantic Engine v1
4.1 Оценка состояния системы
SE v1 может:

построить список сущностей по IMU

выделить:

TOP-линии

TOP-идеи

TOP-миры

найти “мёртвые зоны” (низкий IMU)

4.2 Предложения автору
Типы предложений:

“Эта линия получила высокий IMU, возможно, стоит развить её в мир.”

“Эта идея долго существует, но без связей — либо развить, либо отправить в архив.”

“Этот мир влияет на множество линий — стоит оформить его подробнее.”

“Здесь сформировалась молекула с высоким напряжением — возможна новая линия или конфликт.”

4.3 Работа с вниманием
С учётом Sense Economy:

не предлагать слишком много направлений сразу

давать 2–4 контекстных варианта

учитывать эмоциональное состояние линий (если помечено в фронтматтер)

5. Ограничения v1
работает на уровне оценки и предложений, а не автогенерации структуры

не изменяет файлы сам по себе

уважает архитектуру и волю автора

не вводит IMU как поле везде (это задача для Codex/агентов)

6. Эволюция Semantic Engine
Планы развития:

v1 — оценка + предложения

v2 — полуавтоматические операции (draft новых файлов, индексов)

v3 — адаптивная стратегия развития author-mind

v4 — частично автономное “смысловое поведение” (только с согласия автора)

7. Notes for LLM
использовать IMU как внутренний сигнальный слой, даже если численно он не записан

держать в голове: задача не “максимизировать IMU”, а поддерживать живую, разнообразную систему

предложения автору должны быть мягкими и вариативными
