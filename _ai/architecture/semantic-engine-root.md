# Semantic Engine — Root Spec (v1)

Этот документ описывает, **как author-mind принимает решения о смыслах**:  
что развивать, что связывать, что экспортировать, что оставлять как сырьё.

Semantic Engine v1 — это не кодовый модуль, а:

- набор **правил**,  
- соглашения о **фронтматтере**,  
- **процессы**, по которым LLM и автор действуют внутри системы.

Позже это может быть автоматизировано Codex/скриптами, но смысловая модель — здесь.

---

## 1. Цели Semantic Engine v1

1. **Не дать смыслу раствориться в объёме.**  
   При росте author-mind должна оставаться видимой структура важного.

2. **Поддерживать здравый баланс между архитектурой и содержанием.**  
   Не допустить, чтобы мы бесконечно достраивали форму, забывая про смысл.

3. **Подсказывать направления развития.**  
   Показывать, какие линии, идеи, stories и проекты “просятся” вперёд.

4. **Готовить почву для автоматизации в v2+.**  
   Так, чтобы Codex мог зайти в уже сформированную смысловую модель.

---

## 2. С какими сущностями работает движок

Semantic Engine рассматривает такие типы сущностей:

- **ideas/** — атомарные фрагменты сознания, интуиции, потоки (сырьё).  
- **lines/** — зрелые философские / художественные линии.  
- **lines_summaries/** — их сжатые формы (машинный / оркестровый формат).  
- **stories/** — авторская линия Stories (эпизоды).  
- **projects/** — описания внешних репозиториев / проектов.  
- **exports/** — то, что уходит наружу (в другие репо, в публикации).  
- **worlds/** — миры, сеттинги, мифологии.  
- **aesthetics/** — художественные правила и стилистика.  
- **emotions/** — карты эмоциональных тонов.

В v1 Semantic Engine работает в основном с:

- `ideas/`  
- `lines/`  
- `stories/`  
- `projects/`  
- `exports/`

---

## 3. Сигналы, которые он учитывает

На уровне v1 мы фиксируем несколько ключевых сигналов:

- **IMU** — внутренняя смысловая значимость сущности (идея/линия/сторис).  
- **Связи** — количество и качество ссылок (из index-файлов, ссылок в тексте).  
- **Статус** — `draft` / `active` / `stable` / `archived`.  
- **Роль** — `core`, `support`, `experimental`, `meta`.  
- **Research Hooks** — наличие зацепок для будущих исследований.  
- **Export potential** — пригодность к экспорту (в Stories, в проекты, в статьи).

В дальнейшем сюда могут добавиться:

- внешние метрики (AEU/PEU/SIU),  
- внимание автора (частота обращений),  
- связи с реальными событиями (по imports).

## 4. Минимальные операции Semantic Engine v1

В v1 движок не “работает сам по таймеру”.  
Он **вызывается тобой или LLM** в момент работы.

Он должен уметь:

1. **Оценить сущность.**

   Пример:  
   > “Оцени эту идею/линию/сторис по IMU и предложи, что с ней делать”.

2. **Предложить маршрут.**

   Например:

   - `idea → line`  
   - `line → story`  
   - `story → export`  
   - или наоборот: `import → story → line`.

3. **Подсветить горячие точки.**

   - какие линии “висят недописанными”,  
   - какие идеи давно ждут развития,  
   - какие stories напрашиваются на оформление.

4. **Обнаружить пробелы.**

   - “у нас есть философия X и архитектура Y, но нет связующей линии”,  
   - “есть Stories, но нет общей эстетики”,  
   - “есть value-model, но не прописаны практические сценарии”.

5. **Давать мягкие предупреждения.**

   - “архитектурных файлов становится слишком много относительно содержания”,  
   - “эта линия повторяет уже имеющуюся структуру, возможно, стоит слить”.

## 5. Формат фронтматтера для работы с движком

Для сущностей, с которыми работает Semantic Engine, важно добавить в фронтматтер:

```yaml
imu: <float>               # оценка внутренней значимости (примерно, 0.0–10.0)
role: core|support|meta|experimental
status: draft|active|stable|archived

engine_notes: ""           # короткие пометки для Semantic Engine
export_potential: low|medium|high
links_hint:                # необязательные подсказки, во что это должно связаться
  - type: line
    id: line-000-metathoughts
  - type: project
    id: vovaipetrova-core
